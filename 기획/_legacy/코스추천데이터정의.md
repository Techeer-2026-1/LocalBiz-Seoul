# 로컬 코스 추천 파이프라인 설계

## 1. 개요
단일 장소 분석을 넘어 '코스(Course)' 단위의 추천을 제공하기 위해, 기존 정적 데이터(LLM 스코어)에 공간(Spatial) 및 시간(Temporal) 데이터를 결합하는 동적 쿼리 기반 아키텍처입니다.

## 2. 필수 추가 데이터 (DB 스키마 요구사항)
기존 `place_analysis` (LLM 분석 결과) 테이블과 조인될 장소 메타데이터 테이블에 다음 컬럼이 필수적으로 요구됩니다.

* **좌표 데이터 (위도/경도):** 공간 검색(반경 계산)을 위한 핵심 데이터. (FLOAT/DECIMAL 또는 Spatial Data Type)
* **카테고리 (분류):** 식당, 카페, 문화공간, 야외 등 코스의 흐름(Sequence)을 제어하기 위한 기준 값.
* **영업시간 및 휴무일:** 사용자가 방문하는 시점에 장소가 오픈 상태인지 검증하기 위한 필수 데이터.
* **예상 체류 시간:** 코스 전체 소요 시간을 계산하기 위한 기준값 (예: 식당 1.5시간, 카페 1시간).

## 3. 코스 추천 조합 알고리즘 (백엔드 로직)
미리 모든 코스의 경우의 수를 생성해 두는 것은 비효율적이므로, 사용자 요청 시 백엔드에서 위치 기반 SQL 쿼리를 통해 실시간으로 조합(Dynamic Assembly)합니다.

### 1단계: 시작점 (Anchor) 선정
* 사용자가 현재 위치를 전송하거나 특정 장소를 '찜'했을 때, 해당 좌표를 기준점(Anchor)으로 설정합니다.

### 2단계: 반경 검색 (Geospatial Filtering)
* 공간 쿼리 함수(예: PostgreSQL `PostGIS`, MySQL `ST_Distance_Sphere`)를 사용하여 기준점으로부터 반경 N 킬로미터(예: 500m ~ 1km) 이내의 장소만 1차 필터링합니다.

### 3단계: 카테고리 및 테마 필터링 (DB 조인)
* **흐름 제어:** 1차 장소가 '식당'이면 2차 장소는 '카페' 또는 '문화공간'으로 제한합니다.
* **테마 적용:** 1단계 파이프라인에서 적재한 LLM 스코어를 `WHERE` 절에 포함하여 타겟팅합니다.
    * *예: 반경 1km 이내 + 카테고리='카페' + LLM_분위기_점수 >= 4.0*

## 4. 상황별 데이터 추천 쿼리 시나리오 (예시)

* **가성비 데이트 코스:**
    * 조건: 반경 1km 내 [식당 -> 문화공간 -> 카페] 순서.
    * 필터: LLM 분석 스코어 중 `Price(가격)` 점수가 높은 장소 우선 정렬.
* **맛과 서비스 중심 코스:**
    * 조건: 반경 1km 내 [식당 -> 카페] 순서.
    * 필터: LLM 분석 스코어 중 `Taste(맛)` >= 4.5 AND `Service(서비스)` >= 4.0 인 장소.

## 5. 아키텍처 결론
코스 추천에 LLM을 실시간으로 개입시키지 않습니다. LLM은 각 장소의 속성을 수치화(Scoring)하여 DB에 적재하는 역할만 수행하며, 실제 코스 조합은 백엔드 서버의 RDBMS 공간 쿼리(Spatial Query)를 통해 1초 이내에 처리되도록 설계합니다.
