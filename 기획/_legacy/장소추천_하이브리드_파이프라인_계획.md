# 장소 추천 — 하이브리드 파이프라인 구현 계획

작성일: 2026-04-01 | 작성: 이정

---

## 1. 배경

### 1.1 해결하려는 문제

사용자가 "반경 1km 이내에 카공하기 좋은 카페 추천해줘"라고 요청할 때:

- **"반경 1km", "카페"** → 정형 조건. RDB 쿼리로 처리 가능
- **"카공하기 좋은"** → 비정형 조건. DB 컬럼만으로는 판별 불가

비정형 조건을 처리하려면 리뷰 데이터 기반 의미 매칭이 필요하지만, 현재 리뷰 데이터가 810건 장소 중 2건만 존재한다.

### 1.2 핵심 아이디어

**실시간 리뷰 수집 대신, LLM이 쿼리를 확장하여 검색 엔진에 의미 매칭을 위임한다.**

```
사용자: "카공하기 좋은 카페"
    ↓ LLM 쿼리 확장
확장 검색어: "콘센트 카페", "넓은 테이블 스터디 카페", "작업하기 좋은 카페"
    ↓ 확장 검색어로 API + DB 병렬 검색
    ↓ 후보가 이미 필터링된 상태
    ↓ 메타 필터 (위치, 영업시간) + 리랭킹
    ↓ 상위 N건 반환
```

Google Places Text Search는 내부적으로 리뷰 데이터를 활용한 의미 매칭을 수행하므로, **쿼리를 잘 만들어주면 리뷰 수집 없이도 비정형 조건 반영이 가능**하다.

## 2. 현재 구조 (AS-IS)

### 2.1 place_recommend_node 흐름

```
사용자 쿼리
    ↓
① LLM 조건 추출 (Gemini JSON mode)
    → category, location, conditions, queries[3], limit
    ↓
② 다중 쿼리 병렬 Google Places Text Search
    → queries[3] × 5건 = 15건 → 중복 제거 → 평점순 limit건
    ↓
③ 장소별 네이버 블로그 + Google 리뷰 병렬 수집
    → 장소당 Naver 10건(snippet) + Google 5건
    ↓
④ 조건 키워드 기반 리뷰 리랭킹 (_rerank: 단순 키워드 in 매칭)
    ↓
⑤ 장소별 LLM 추천 이유 생성 (text_stream)
    ↓
⑥ 응답: place 카드 + 추천 이유 + references
```

### 2.2 문제점

| 문제 | 상세 |
|------|------|
| DB 미활용 | 810건 places 테이블을 전혀 사용하지 않음. 매번 Google API 의존 |
| 위치 필터 없음 | `user_location`을 받고 있지만 반경 검색 미구현 |
| 리뷰 리랭킹 조잡 | `_rerank`가 단순 `keyword in text` 카운트. 의미 매칭 아님 |
| 쿼리 확장 부족 | `queries[3]`이 조건 분배 수준. "카공" → "콘센트" 같은 동의어 확장 없음 |
| place_analysis 미활용 | 배치 분석 결과(6개 지표, 키워드, 요약)가 있어도 추천에 반영 안 됨 |

### 3.2 쿼리 확장 프롬프트 설계

```
시스템:
사용자의 장소 추천 요청에서 조건을 추출하고, 검색 엔진에 최적화된 쿼리로 확장하세요.

JSON으로만 응답:
{
  "category": "카테고리 (카페, 맛집, 헬스장, 미용실, 공원 등)",
  "category_code": "DB 카테고리 코드 (cafe, restaurant, culture, park, shopping, other)",
  "location": "지역명 (없으면 '서울')",
  "radius_km": 반경 km (언급 없으면 null),
  "conditions": ["원본 비정형 조건들"],
  "expanded_queries": [
    "동의어/구체화된 검색어 1 (예: 콘센트 카페 + 지역)",
    "동의어/구체화된 검색어 2 (예: 스터디 카페 + 지역)",
    "동의어/구체화된 검색어 3 (예: 넓은 테이블 조용한 카페 + 지역)"
  ],
  "analysis_weights": {
    "taste": 0~1,
    "service": 0~1,
    "atmosphere": 0~1,
    "value": 0~1,
    "cleanliness": 0~1,
    "accessibility": 0~1
  },
  "limit": 숫자 (기본 5)
}

expanded_queries 작성 규칙:
- 사용자의 비정형 조건을 Google/Naver 검색이 잘 이해할 구체적 키워드로 변환
- 동의어, 유사 표현, 세부 속성으로 확장
- 각 쿼리에 location을 포함시킬 것
- 예시:
  "카공" → "콘센트 카페", "스터디 카페", "작업하기 좋은 조용한 카페"
  "데이트" → "분위기 좋은 레스토랑", "루프탑 레스토랑", "야경 맛집"
  "가성비" → "가성비 맛집", "만원 이하 점심", "저렴한 맛집"
  "아이랑" → "키즈카페", "놀이방 있는 식당", "유아 동반 맛집"

analysis_weights 작성 규칙:
- 비정형 조건과 가장 관련 높은 지표에 높은 가중치
- 합이 1.0이 되도록 정규화
- 예시:
  "카공" → atmosphere 0.35, accessibility 0.25, value 0.25, service 0.15
  "데이트" → atmosphere 0.5, taste 0.2, service 0.2, cleanliness 0.1
  "가성비" → value 0.5, taste 0.3, service 0.1, atmosphere 0.1
```

## 4. DB 검색 함수 설계

### 4.1 Haversine 반경 검색 (PostGIS 없이)

```python
async def search_places_db(
    category_code: str,
    lat: float | None,
    lng: float | None,
    radius_km: float | None,
    keywords: list[str] | None = None,
    limit: int = 10,
) -> list[dict]:
    """
    PostgreSQL places 테이블에서 조건 검색.
    PostGIS 없이 Haversine 공식으로 반경 필터링.
    """
    query = """
        SELECT *,
            ( 6371 * acos( cos(radians(%s)) * cos(radians(lat))
              * cos(radians(lng) - radians(%s))
              + sin(radians(%s)) * sin(radians(lat)) ) ) AS distance_km
        FROM places
        WHERE category = %s
    """
    params = [lat, lng, lat, category_code]

    if radius_km:
        query += " HAVING distance_km < %s"
        params.append(radius_km)

    query += " ORDER BY distance_km ASC LIMIT %s"
    params.append(limit)

    # → 후보 반환
```

## 5. 리랭킹 알고리즘

### 5.1 종합 스코어 산출

```python
def compute_score(place: dict, analysis: dict | None, weights: dict, expanded_keywords: list[str]) -> float:
    """
    score = 0.3 × normalized_rating        # Google 평점 (0~1)
           + 0.3 × weighted_analysis_score  # place_analysis 가중 지표 (0~1)
           + 0.2 × keyword_match_bonus      # 키워드 매칭 보너스 (0~1)
           + 0.2 × popularity_score         # 리뷰 수 로그 정규화 (0~1)
    """
```

### 5.2 각 항목 상세

| 항목 | 계산 방법 | place_analysis 없을 때 |
|------|----------|---------------------|
| normalized_rating | `rating / 5.0` | Google 평점만 사용 |
| weighted_analysis_score | `Σ(weight_i × score_i) / 5.0` | **0으로 처리**, 나머지 항목에 가중치 재분배 |
| keyword_match_bonus | 확장 키워드 토큰 ∩ analysis.keywords / len(확장 키워드) | **0으로 처리** |
| popularity_score | `log10(review_count + 1) / log10(max_review_count + 1)` | Google user_ratings_total 사용 |

### 5.3 place_analysis 없는 장소의 처리

**분석 데이터가 없는 장소가 대다수(810건 중 2건만 존재)이므로:**

- 분석 데이터 없으면 해당 항목 0 → **가중치를 rating + popularity에 재분배**
- 실질적으로 `0.5 × rating + 0.3 × popularity + 0.2 × API_source_bonus`
- API 결과(Google Places)는 이미 비정형 조건을 반영하므로, API 출처 장소에 source bonus 부여

```python
# place_analysis 유무에 따른 가중치 동적 조정
if analysis:
    W = {"rating": 0.3, "analysis": 0.3, "keyword": 0.2, "popularity": 0.2}
else:
    W = {"rating": 0.5, "analysis": 0.0, "keyword": 0.0, "popularity": 0.3, "source": 0.2}
    # source: Google API 결과면 0.8, DB 단독이면 0.3
    # → 쿼리 확장으로 검색된 API 결과에 가산점
```
