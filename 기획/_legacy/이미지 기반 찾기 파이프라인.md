# 데이터 추가 & 이미지 기반 장소 찾기 파이프라인 보고서 
> 작성 기준: 2026-03-31
## 0.길찾기 , 지도 표시 , 예약 연동 -> 데이터 추가 불필요




## 1. 기능 개요

사용자가 사진을 올리면 두 가지를 해주는 기능.

- **장소 식별**: "여기 어디야?" → 해당 장소 찾아주기
- **유사 장소 추천**: "이런 분위기 추천해줘" → 비슷한 곳 찾아주기

정확한 장소 식별이 안 되면 유사 장소 추천으로 땜빵하는 구조.

---

## 2. 전체 파이프라인

```
사진 업로드
    ↓
[Step 1] OCR — 텍스트 추출
    ├── 텍스트 없음 → Step 3으로
    └── 텍스트 있음
            ↓
        [Step 2] 텍스트 개수 판단
            ├── 1개 → Step 4로 (DB 검색)
            └── 여러 개
                    ↓
                [Step 2-A] Gemini Vision으로 주인공 판단
                    ├── 판단 성공 → Step 4로
                    └── 판단 실패 → 사용자에게 선택지 제시
                            "어디를 찾으시나요? ① 스타벅스 ② 맥도날드"
                            사용자 선택 → Step 4로
    ↓
[Step 3] Gemini Vision — 텍스트 없을 때
    ├── 장소 타입 분류 (음식점/관광지/쇼핑/공원/숙박)
    ├── 분위기 키워드 추출
    └── 랜드마크 여부 판단
            ├── 랜드마크 → OpenSearch 텍스트 검색 → 장소 식별
            └── 일반 장소 → Step 6 (유사 추천)으로
    ↓
[Step 4] DB 검색
    ├── 있음
    │   ├── 단일 장소 → Step 5로
    │   └── 체인점 → GPS 기준 가까운 지점 → Step 5로
    └── 없음 → Google Places API 2차 검색
            ├── 있음 → Step 5로
            └── 없음 → Step 6 (유사 추천)으로
    ↓
[Step 5] 질문 의도 파악 (장소 식별 성공한 경우)
    ├── "여기 어디야?" → 장소 정보 반환
    ├── "영업시간/예약?" → Google Places 실시간 조회
    ├── "근처 맛집?" → 코스 추천 파이프라인 연결
    └── "예약해줘" → 예약 파이프라인 연결
    ↓
[Step 6] 유사 장소 추천 (식별 실패 or 추천 요청)
    ├── 키워드 추출 성공
    │   ├── "추천해줘" → OpenSearch 유사도 검색 → Top 5
    │   ├── "근처에서" → 유사도 검색 + GPS PostGIS 필터
    │   └── "강남에서" → 유사도 검색 + district 필터
    └── 키워드 추출 실패 (사진 불명확)
            → "사진이 불명확해요. 어떤 분위기를 원하세요?" 되물어보기
```

## 3. 질문 경우의 수

| 케이스 | 예시 질문 | 처리 방식 |
|------|------|------|
| 장소 식별 | "여기 어디야?" | OCR + Gemini Vision → DB 검색 |
| 유사 장소 추천 | "이런 분위기 추천해줘" | 키워드 → OpenSearch 유사도 |
| 위치 기반 유사 추천 | "이런 느낌 근처에 있어?" | 키워드 + GPS PostGIS 필터 |
| 지역 기반 유사 추천 | "이런 느낌 강남에서 찾아줘" | 키워드 + district 필터 |
| 장소 정보 조회 | "여기 영업시간?" | 식별 후 Google Places 조회 |
| 코스 추천 | "여기 근처 맛집 추천해줘" | 식별 후 코스 파이프라인 연결 |
| 예약 | "여기 예약해줘" | 식별 후 예약 파이프라인 연결 |

## 4. 장소 타입별 특이사항

### 음식점/카페
- OCR 간판 → 식별 우선
- 실패 시 분위기 키워드로 유사 추천
- 리뷰 데이터 있으면 추천 품질 올라감
- 체인점 많음 → GPS로 가까운 지점 찾기 중요

### 관광지/랜드마크
- Gemini Vision이 바로 알아볼 가능성 높음
- 식별 성공 시 영업시간/입장료/예약 안내
- "비슷한 관광지 추천" 요청은 드물어서 낮은 우선순위
- TourAPI 이미지 무료로 활용 가능

### 쇼핑몰/백화점
- 간판 크고 명확해서 OCR 성공률 높음
- 체인점 많아서 GPS로 가까운 지점 찾기 중요
- 영업시간 질문이 많음

### 공원/자연
- 간판 거의 없음 → OCR 스킵하고 Gemini Vision 바로
- "이런 공원 추천해줘" 질문 多
- GPS 필터 중요 (근처 공원 찾기)

### 숙박
- 호텔 간판 OCR 성공률 높음
- 식별 성공 시 예약 파이프라인 바로 연결
- "이런 분위기 호텔 추천" 유사도 검색 가능
- 체크인/체크아웃 날짜 수집 필요 (멀티턴 대화)

## 5. 엣지 케이스

### 간판 여러 개인 경우
```
OCR → ["스타벅스", "맥도날드", "올리브영"] 여러 개 추출
    ↓
1차: Gemini Vision으로 주인공 판단 (이미지 맥락 보고)
    ↓
판단 실패 시
2차: 사용자에게 선택지 제시
    "어디를 찾으시나요?"
    ① 스타벅스 ② 맥도날드 ③ 올리브영
```

### 사진이 불명확한 경우
```
Gemini Vision → 키워드 추출 실패
"카페, 의자, 테이블" 같은 너무 일반적인 키워드
    ↓
"사진이 불명확해요. 어떤 분위기를 원하세요?" 되물어보기
    ↓
텍스트로 보완 받아서 재검색
```

### DB에 없는 장소인 경우
```
OCR → 장소명 추출
    ↓
DB 검색 → 없음
    ↓
Google Places API 2차 검색
    ├── 있음 → 장소 정보 반환
    └── 없음 → 유사 장소 추천으로 땜빵
```

### 체인점인 경우
```
OCR → "스타벅스" 추출
    ↓
DB → 스타벅스 여러 개
    ↓
GPS 기준 제일 가까운 지점 반환
    "가장 가까운 스타벅스는 강남점이에요 (도보 5분)"
```

## 6. 고려해야 할 사항

### 데이터 관련
- 장소별 분위기/키워드 설명이 DB에 없으면 유사도 검색 품질 낮음
- 리뷰 데이터 선행 적재 필요 (네이버 검색 API or 크롤링)
- 이미지 자체 저장은 S3 없이 URL만 저장 (Google Places Photo URL / TourAPI)
- 서울 전체 이미지 적재는 비용 문제로 관광지/랜드마크 한정 권장

### 정확도 관련
- 일반 식당 정확한 식별은 현실적으로 어려움 → 유사 추천으로 땜빵
- 랜드마크는 Gemini Vision 정확도 높음
- OCR은 간판이 잘 찍혔을 때만 정확
- 야간/역광 사진은 OCR 및 이미지 분석 정확도 낮아짐

### 비용 관련
- OCR (easyocr): 무료
- Gemini Vision API: 유료 (팀 키 필요)
- Google Places API: 유료 (월 $200 무료 크레딧)
- 리뷰 요약 Claude Haiku: 배치 API 기준 약 $30 (장소당 상위 3개 리뷰)

### UX 관련
- 식별 실패 시 바로 에러 말고 유사 추천으로 자연스럽게 전환
- 간판 여러 개일 때 사용자 선택 UX 필요
- 되물어보기는 최대 1회로 제한 (너무 많으면 불편)

## 10. 결론

Gemini Vision API 키만 생기면 OCR부터 바로 POC 가능.
리뷰 데이터 없어도 카테고리 기반으로 일단 돌아가게 만들고 나중에 고도화.
정확한 장소 식별보다 유사 장소 추천에 집중하는 게 현실적.
