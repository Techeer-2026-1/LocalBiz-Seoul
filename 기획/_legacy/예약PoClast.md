# 예약 연동 POC 기획안 v2

> 작성 기준: 2026-03-25
> 핵심 전제: AI가 직접 예약 완료는 불가. 모든 예약은 딥링크(해당 예약 페이지 연결) 방식.

---
## 0. 우선사항
1. 숙박,식당 등으로 도메인 좁히기
2. api 연동 더 찾아보기

## 1. 핵심 원칙

- **직접 예약 완료 불가** — 결제/개인정보 입력은 사용자가 직접 해야 함
- **딥링크 방식** — AI는 적절한 예약 페이지로 연결만 해줌
- **딥링크 URL은 AI가 자동 생성** — 장소명만 있으면 생성 가능, DB에 미리 저장 불필요
- **예약 가능 여부/링크는 실시간 조회** — DB에 저장하면 데이터 변경 시 업데이트 문제 발생
- **DB에 없는 장소 바로 컷 금지** — Google Places API로 2차 검색 후 판단

---

## 2. 데이터 소스 전략

예약 관련 정보를 어디서 가져오는지가 핵심.

| 정보 | 출처 | 방식 |
|------|------|------|
| 장소 기본 정보 (이름, 주소, 카테고리) | CSV → PostgreSQL DB | 배치 ETL로 사전 적재 |
| 예약 링크, 영업시간, 전화번호 | Google Places API | 사용자 요청 시 실시간 조회 |
| 공연/행사 잔여석, 예매 링크 | KOPIS API | 사용자 요청 시 실시간 조회 |
| 공공시설 예약 가능 여부 | 서울시 공공서비스예약 API | 사용자 요청 시 실시간 조회 |
| 분위기/리뷰 등 비정형 | OpenSearch 벡터 검색 | 배치 임베딩 후 검색 |

**핵심:** 예약 가능 여부/링크는 DB에 저장하지 않고 실시간 API 조회가 원칙.
Google Places API에 없는 소규모 식당 등은 전화번호만 안내하는 fallback으로 처리.

## 3. 공통 앞단 파이프라인

```
사용자: "OO 예약해줘"
    ↓
[의도 분류] BOOKING 감지
    ↓
[범위 검증] 서울 로컬 외 요청 컷
    예) 비행기, 해외호텔, 기차 등
    → "저는 서울 로컬 장소 예약만 도와드릴 수 있어요"
    ↓
[장소 확인 - 2단계]
    1차: PostgreSQL DB 검색
    2차: DB에 없으면 → Google Places API 실시간 검색
    그래도 없으면 → "해당 장소를 찾을 수 없어요"
    ↓
[카테고리 분류]
    음식점 / 행사·공연 / 공공시설 / 숙박 / 관광지
    ↓
카테고리별 파이프라인으로 분기
```

## 4. 카테고리별 파이프라인

### 4-1. 음식점

**데이터 소스:** Google Places API 실시간 조회

```
[Google Places API 실시간 조회]
    (예약링크, 전화번호, 영업시간, 현재 영업 여부)
    ↓
[예약 방식 판단]
    ├── 네이버 예약 링크 있음 → 네이버 예약 딥링크
    ├── 카카오맵 링크 있음   → 카카오맵 예약 딥링크
    ├── 구글 예약 있음       → 구글 예약 딥링크
    ├── 전화번호만 있음      → "전화 예약만 가능해요 02-XXX"
    └── 정보 없음            → "예약 정보를 찾을 수 없어요"
```

**딥링크 예시:**
- 네이버: `https://booking.naver.com/search?query=장소명`
- 카카오: `https://place.map.kakao.com/search?q=장소명`
- 구글: `https://search.google.com/local/reservations?query=장소명`

**현실적 한계:**
- Google Places에 등록 안 된 소규모 식당 → 전화번호 안내 fallback
- 네이버/카카오 예약 API 비공개 → 딥링크만 가능 (실시간 잔여석 확인 불가)

---

### 4-2. 행사·공연

**데이터 소스:** KOPIS API + 서울 문화행사 API 실시간 조회

```
[KOPIS API / 서울 문화행사 API 실시간 조회]
    (잔여석, 요금, 예매 링크, 공연 기간)
    ↓
[예약 방식 판단]
    ├── 잔여석 있음   → 예매 딥링크 (인터파크/KOPIS/티켓링크)
    ├── 무료 행사     → "무료 행사예요, 현장 입장 가능해요"
    ├── 예약 마감     → "예약이 마감됐어요"
    └── 정보 없음     → "예매 정보를 찾을 수 없어요"
```

**딥링크 예시:**
- KOPIS: `http://www.kopis.or.kr/search?query=공연명`
- 인터파크: `https://ticket.interpark.com/search?query=공연명`

**현실적 한계:**
- 인터파크/티켓링크 예매 API 비공개 → 딥링크만 가능
- KOPIS는 공연 정보 조회만 가능, 실제 예매는 외부 사이트로 연결

---

### 4-3. 공공시설

**데이터 소스:** 서울시 공공서비스예약 API 실시간 조회

```
[서울시 공공서비스예약 API 실시간 조회]
    (사전예약 필요 여부, 잔여석, 예약 가능 날짜)
    ↓
[예약 방식 판단]
    ├── 사전 예약 필요 + 잔여 있음 → 예약 페이지 딥링크
    ├── 현장 입장 가능             → "현장 방문하시면 돼요"
    ├── 예약 마감                  → "현재 예약이 마감됐어요"
    └── API 조회 불가              → 구청 예약 페이지 안내
```

**딥링크:** `https://yeyak.seoul.go.kr/search?keyword=시설명`

**왜 DB 저장 안 하는가:**
사전예약 필요 여부, 잔여석은 수시로 바뀌므로 DB 저장 시 업데이트 문제 발생.
실시간 API 조회가 유일한 현실적 방법.

---

### 4-4. 숙박

**데이터 소스:** Google Places API 실시간 조회 + 멀티턴 대화로 날짜 수집

```
[멀티턴 대화로 조건 수집]
    챗봇: "체크인이 언제예요?"
    사용자: "3월 28일"
    챗봇: "체크아웃은요?"
    사용자: "3월 29일"
    ↓
[Google Places API 실시간 조회]
    (전화번호, 평점, 공식 홈페이지)
    ↓
[예약 채널 판단]
    ├── 공식 홈페이지 있음 → 날짜 파라미터 포함해서 직접 링크
    ├── 야놀자 검색 결과   → 날짜 포함 딥링크
    ├── 여기어때 검색 결과 → 날짜 포함 딥링크
    └── 전화만 있음        → "전화 예약만 가능해요"
```

**딥링크 예시:**
- 야놀자: `https://www.yanolja.com/search?checkIn=2026-03-28&checkOut=2026-03-29&keyword=호텔명`
- 여기어때: `https://www.goodchoice.kr/search?keyword=호텔명`

**현실적 한계:**
- 야놀자/여기어때 API 비공개 → 딥링크만 가능 (실시간 객실 잔여 확인 불가)
- 멀티턴 대화 필요 → LangGraph Checkpointer로 대화 컨텍스트 유지 필요

---

### 4-5. 관광지

**데이터 소스:** Google Places API 실시간 조회

```
[Google Places API 실시간 조회]
    (영업시간, 입장료, 예약 링크, 임시 휴관 여부)
    ↓
[입장 방식 판단]
    ├── 사전 예약 필요 → 예약 페이지 딥링크 + 영업시간/입장료 안내
    ├── 현장 구매 가능 → "현장 구매 가능해요" + 영업시간/입장료 안내
    ├── 무료 입장      → "무료예요, 바로 방문하세요!" + 영업시간
    └── 임시 휴관      → "현재 휴관 중이에요"
```

**예시 응답:**
```
경복궁 안내
- 영업시간: 09:00 ~ 18:00 (화요일 휴관)
- 입장료: 성인 3,000원
- 사전 예약: 필요없음, 현장 구매 가능
- 예약 링크: [문화재청 예약 페이지]
```

## 5. 전체 흐름 요약

```
BOOKING 의도 감지
    ↓
범위 검증 (서울 로컬만)
    ↓
장소 확인
    1차: PostgreSQL DB
    2차: Google Places API
    ↓
카테고리 분류
    ↓
음식점    → Google Places 실시간      → 네이버/카카오/구글/전화
행사·공연 → KOPIS API 실시간          → 인터파크/KOPIS 딥링크
공공시설  → 공공서비스예약 API 실시간  → 예약 딥링크
숙박      → 멀티턴 대화 + Google Places → 야놀자/여기어때
관광지    → Google Places 실시간      → 예약링크/영업시간 안내
    ↓
사용자에게 예약 링크 or 전화번호 or 안내 전달
```

## 6. 사용 API 정리

| API | 용도 | 비용 | 비고 |
|-----|------|------|------|
| Google Places API | 예약링크, 영업시간, 전화번호 실시간 조회 | 유료 (월 $200 무료 크레딧) | 핵심 API |
| KOPIS API | 공연/행사 정보, 잔여석 조회 | 무료 | 공공 API |
| 서울 문화행사 API | 서울 행사 정보 조회 | 무료 | 공공 API |
| 서울시 공공서비스예약 API | 공공시설 예약 가능 여부 실시간 조회 | 무료 | 공공 API |

## 7. 한계점

| 한계 | 이유 | 대안 |
|------|------|------|
| 직접 예약 완료 불가 | 결제/개인정보는 사용자가 직접 | 딥링크로 해당 페이지 연결 |
| 네이버/카카오 예약 API 없음 | 비공개 API | 딥링크 + 전화번호 안내 |
| 야놀자/여기어때 API 없음 | 비공개 API | 딥링크로 검색 결과 연결 |
| 인터파크/티켓링크 예매 API 없음 | 비공개 API | KOPIS 딥링크로 연결 |
| 소규모 식당 정보 부족 | Google Places 미등록 | 전화번호 안내 fallback |
| 숙박 실시간 객실 잔여 확인 불가 | 야놀자/여기어때 API 비공개 | 딥링크에 날짜 파라미터 포함 |

## 8. POC 우선순위

| 순위 | 카테고리 | 이유 |
|------|------|------|
| 1 | 음식점 | CSV 데이터 가장 많음, Google Places 커버리지 높음 |
| 2 | 행사·공연 | KOPIS API 무료, 데이터 풍부 |
| 3 | 관광지 | Google Places로 대부분 커버 가능 |
| 4 | 공공시설 | 서울시 API 무료, 구현 비교적 단순 |
| 5 | 숙박 | 멀티턴 대화 구현 필요, 난이도 높음 |
