# 1. 서비스 개요

## 1.1 프로젝트 소개

LocalBiz Intelligence는 **서울 로컬 라이프 AI 챗봇**이다. 사용자가 자연어로 질문하면 서울의 장소, 행사, 맛집, 카페 등에 대해 AI가 실시간으로 검색·추천·분석하여 응답한다.

핵심 차별점:

- **하이브리드 검색**: 정형(PostgreSQL + PostGIS) + 비정형(OpenSearch 768d k-NN) 동시 처리
- **LLM 기반 분석**: 리뷰 6개 지표 채점, 코스 추천, 비용 견적 등
- **실시간 스트리밍**: WebSocket 기반 블록 순서 응답
- **서울 특화**: 151만 장소 + 7천 행사 + 29만 생활인구 데이터

## 1.2 팀 구성

| --- | --- | --- |

```
  ├─→ Gemini API: LLM 응답 생성

  ├─→ Naver Blog API: 블로그 리뷰/가격
  └─→ Google Calendar API: 일정 추가
```

# 3. 기능 명세 요약

> 상세는 [📋 기능 명세서](https://www.notion.so/4669814b7a624c29b5422a85efcda2b1?pvs=21) 참조
> 

## 3.1 Phase 1 — 핵심 기능

**대화 기본:**

의도 분류, 쿼리 전처리(모든 검색 공통), 세션 유지 + 전체 대화 이력 영속 저장(이원화), 대화 스트리밍, 처리 상태 표시, 응답 중단(Interrupt), 새 대화 시작, 대화 삭제, 응답 재생성(Retry), 에러 처리/재시도, 대화 제목 자동생성/수정

**도메인 기능:**

장소 검색(PLACE_SEARCH), 장소 추천(PLACE_RECOMMEND) + 사유 표시, 행사 검색(EVENT_SEARCH, 정형+비정형), 코스 추천(COURSE_PLAN) + 경로 표시, 예약 연동(BOOKING, 딥링크), 일정 추가(CALENDAR), 지도 마커

**사용자 관리:**

회원가입/로그인(이메일+비밀번호 + Google OAuth), 프로필 수정/로그아웃

## 3.2 Phase 2 — 확장 기능

**도메인:** 리뷰 비교(REVIEW_COMPARE), 비용 견적(COST_ESTIMATE), 혼잡도 분석(CROWDEDNESS), 단일 장소 분석(ANALYSIS), 이미지 검색(IMAGE_SEARCH), 중첩된 요청 대응(복수 intent 순차 실행)

**사용자 기능:** 북마크(생성/목록/필터/이동/삭제, 5종 프리셋 핀), 대화 타임스탬프, 대화 공유(share_token 기반)

## 3.3 Phase 3

AI 응답 피드백 (👍/👎 + 텍스트 코멘트)

## 3.4 ETL

리뷰 배치 분석(Naver Blog + Google Reviews → Gemini Flash → place_analysis), 모니터링(Grafana+Prometheus+Loki+Jaeger → Slack)

## 4.1 설계 원칙

- 대부분의 기능은 **채팅 WS 안에서 처리**
- REST는 채팅 밖에서 독립 호출이 필요한 것만 (인증, 대화 관리, 북마크, 공유)
- WS 응답은 **"블록 순서"**로 정의 — 서버가 여러 프레임을 순서대로 스트리밍

## 4.5 WS 응답 스키마

**Phase 1 intent:**

| intent | 응답 블록 순서 |
| --- | --- |
| GENERAL | intent → text_stream → done |
| PLACE_SEARCH | intent → status → text_stream → places[] → map_markers → done |
| PLACE_RECOMMEND | intent → status → text_stream → places[] → map_markers → references → done |
| EVENT_SEARCH | intent → status → text_stream → events[] → done |
| COURSE_PLAN | intent → status → text_stream → course → map_route → done |
| BOOKING | intent → text_stream(딥링크) → done |
| CALENDAR | intent → text_stream → calendar → done |

**Phase 2 intent:**

| intent | 응답 블록 순서 |
| --- | --- |
| REVIEW_COMPARE | intent → status → text_stream → chart → analysis_sources → done |
| COST_ESTIMATE | intent → status → text_stream → done |
| CROWDEDNESS | intent → status → text_stream → done |
| IMAGE_SEARCH | intent → status → text_stream → place/places[]/disambiguation → done |
| ANALYSIS | intent → status → text_stream → analysis_sources → done |

**WS 제어:**

| 기능 | 동작 |
| --- | --- |
| Interrupt | Client: { type: "cancel" } → Server: done(reason: cancelled) |
| Retry | Client: { type: "retry" } → 마지막 응답 삭제 후 재생성 |
| Status | 노드 전환 시 자동: { type: "status", content: "..." } |
| Error | { type: "error", content: { message, retryable } } |

# 5. ERD 요약

> 상세는 [ERD v6](https://www.notion.so/ERD-73d7a82c52e282e6beed8100c5f81e30?pvs=21) + ERD 상세 설명 보고서(docx) 참조
> 

## 5.1 설계 원칙

- **"공통 기둥 + 유연한 주머니(JSONB)"** 패턴
- PostgreSQL `place_id` = OpenSearch `_id`: application 레벨에서 동일 UUID로 두 DB 연결
- **대화 이력 이원화**: LangGraph checkpoint(압축 가능) + messages(append-only)
- **PK 전략**: OpenSearch 연동 테이블만 UUID, 나머지 BIGINT AUTO_INCREMENT
- **3NF 준수** + 성능 위한 의도적 비정규화 허용

## 5.2 PostgreSQL 테이블 (12개)

| No | 테이블 | PK | 설명 |
| --- | --- | --- | --- |
| 1 | places (장소) | UUID | 서울시 통합 장소 마스터 (151만) |
| 2 | events (행사) | UUID | 행사/축제 (7,301) |
| 3 | place_analysis (장소분석) | UUID | LLM 리뷰 6개 지표 채점 |
| 4 | administrative_districts (행정동) | 자연키 | 서울 427개 행정동 폴리곤 |
| 5 | population_stats (생활인구) | BIGINT | 행정동별 시간대별 인구 (29만) |
| 6 | users (사용자) | BIGINT | 사용자 인증 |
| 7 | conversations (대화) | BIGINT | 채팅 세션 메타 |
| 8 | messages (메시지) | BIGINT | 대화 원본 영속 저장 (append-only) |
| 9 | bookmarks (북마크) | BIGINT | 대화 위치 저장 (5종 핀) |
| 10 | shared_links (공유링크) | BIGINT | 대화 공유 토큰 |
| 11 | feedback (피드백) | BIGINT | AI 응답 피드백 (Phase 3) |
| 12 | langgraph_checkpoints | 복합키 | LangGraph 자동 관리 |

# 6. 주요 설계 결정사항

| 결정 | 내용 |
| --- | --- |
| 즐겨찾기 → 북마크 | 장소 저장이 아닌 "대화 위치 저장" 방식 |
| 대화 이력 이원화 | LangGraph checkpoint(LLM용) + messages(UI용, append-only) |
| REST vs WS | 대부분 WS. REST는 독립 호출 필요한 것만 |
| 쿼리 전처리 공통화 | 모든 검색 기능 공통으로 승격 |
| 행사 검색 구조 | DB 우선 → Naver fallback |
| PK 전략 | OpenSearch 연동 = UUID, 나머지 = BIGINT |
| 장소분석 6개 지표 | 범용 (만족도/접근성/청결도/가성비/분위기/전문성) |
| 정규화 | 3NF + 성능 비정규화 허용 |
| 소프트 삭제 | 테이블 성격별 판단 (append-only/마스터/시계열에는 미적용) |
