## 5. 🚨 사용자 escalate points (자체 결정 불가)

### 5.1 소상공인 상가(상권) 534K vs 기존 places 531K 중복 전략

**문제**: 소상공인 상가(상권) CSV와 기존 places(음식점 인허가 CSV)가 같은 음식점을 다른 출처로 중복 기록할 가능성 높음.

**옵션**:
- (a) 소상공인 상가 전수 INSERT → places ~1.1M row (duplicate 포함, 중복 제거 로직 후속)
- (b) 기존 531K와 주소·상호명 fuzzy match → 중복 row 제거 후 diff만 INSERT
- (c) 기존 531K DROP → 소상공인 상가 전수 fresh load (카테고리 체계 우월)
- (d) 두 source 유지, `source` 컬럼으로 구분, consumer가 중복 허용

**권장**: **(d)** — source 컬럼 이미 존재. deduplication은 하류 consumer(search_agent)가 UNION DISTINCT 또는 bizesId 매칭으로 처리. 데이터 보존성 우선.

**사용자 결정 필요**: 허용 가능한가? 혹은 (a)/(b)/(c) 중 선택?

### 5.2 카테고리 분류표 v0.2 bump 권한

**문제**: v0.1 → v0.2로 대분류 7 → 18 확장은 **PM 권한 결정**. 나는 초안까지만 작성 가능 (`category_table_v0.2_draft.md` 참조).

**요청**: 사용자가 초안 검토 후 `기획/카테고리_분류표.md`로 승격 + v0.2 정식 bump. 별도 commit 필요.

### 5.3 seoul_* 전국 CSV 처리 전략

**문제**: `seoul_음식 807K / seoul_소매쇼핑 646K / seoul_숙박 72K`는 **전국 데이터** (`ctprvnNm != '서울특별시'` 포함). 서울 필터 후에도 기존 places와 중복 가능성 높음.

**옵션**:
- (a) 서울 필터 + 기존 places 중복 제거 후 추가 INSERT
- (b) seoul_* 전면 skip (기존 소스로 충분)
- (c) 선별 적재 (source 다양성 확보용)

**권장**: **(b)** — profile_report §7.2에서 "중복 높음" 명시. 소상공인 상가 정보(5.1)만으로 충분. seoul_* 스킵.

**사용자 결정 필요**: (b) 확정 가능한가?
